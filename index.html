浏览器是怎么工作的

浏览器的主要组成部分

  1. 用户界面-包括地址栏，向前向后按钮，书签菜单等。
  2. 浏览器引擎-查询和操作渲染引擎的接口。
  3. 渲染引擎-作用是显示请求的资源，比如，如果请求的内容是HTML，那么它就负责解析HTML和CSS然后把解析的结果显示到屏幕上。
  4. 网络模块-发送网络请求，比如HTTP请求。
  5. 后端UI-绘制基础元素。它提供一个与平台无关的通用接口，在往下就是操作系统提供的用户接口方法。
  6. JavaScript解释器-解析和执行JavaScript代码。
  7. 数据存储-浏览器需要存各种数据在硬盘上，比如，cookies。

渲染引擎

基本流程

从网络层得到文档后，渲染引擎会先从解析HTML文档开始，把标签翻译为DOM节点，生成一个内容树(content tree)。解析CSS，得到的样式信息和之前的内容树结合生成渲染树(render tree)。渲染树经过布局(layout)处理，即确定每一个节点在屏幕上的位置后，就开始绘制(painting)，生成最终的页面。

处理scirpts和CSS的顺序

Scripts

开发者会期望在解析时遇到标签后，立刻解析和执行scripts（因为script执行可能会改变DOM结构）。同时，会停止解析文档直到script执行完后。如果script是引用外部文件，解析也会停止，直到外部文件被下载和执行完毕。但是开发者可以在script标签上添加defer属性，这样script会在文档解析完后再执行。另外，async属性使得外部script的解析和执行是在一个新的线程上，不停止文档的解析。

解析优化

当执行script，会另起一个线程解析剩余的文档，加载外部引用的的资源。注意的是，它只会解析外部引用文件，如，外部scripts，外部CSS和图片等，不会修改DOM树。

CSS

因为CSS不改变DOM树的结构，看起来没有理由要等CSS解析而停止文档的解析。但会存在一个问题，如果scripts在文档解析阶段需要请求样式信息，如果CSS还没被加载和解析，那么scirpts就会因为得不到结果而产生一系列的问题。看起来极端的情况却是经常出现的。Firefox会阻塞所有scripts当还有CSS正在被加载和解析时。Webkit只当scripts试图访问特定的样式信息时阻塞他们。

webpack使用和配置

开始

准备入口文件

在webpack开始打包之前我们需要一个入口文件，来告诉webpack从哪里开始构建它的依赖图(dependency graph)。例如对于vue单页面应用来说，通常会在这个文件引入整个项目的全局依赖包，如，vue，vue-router,vuex等，还有页面入口，一般为App.vue。然后在这个文件中新建vue实例，挂载到App页面上，并设置路由等。

使用配置文件

虽然我们在命令行中运行webpack命令并提供参数来打包，但对于复杂的启动项来说，使用配置文件可以使我们不需要每次打包都执行很长的一串命令。

基本的配置文件一般命名为webpack.config.js，若要区分开发环境和生产环境，会分开配置，分为一个基本的配置文件webpack.base.conf.js，开发环境下的配置webpack.dev.conf.js，生产环境下的配置webpack.prod.conf.js。当然，文件名可以根据自己的喜好来命名，但应表明文件的用途。

使用NPM script

在package.json中为我们的webpack命令设置别名吧，这样，只需运行npm run dev或者npm run build即可运行或build项目。

管理静态文件

因为webpack只能识别js文件，但可以通过loader来加载非js文件并做一些处理。通常，

  • 使用style-loader和css-loader来加载CSS文件
  • 使用url-loader加载图片，音频，视频，字体等文件。也可使用file-loader，但url-loader会将小于一定比特的文件使用base64编码嵌入到文档中，从而减少网络请求。
  • 若在开发中使用ES6/7语法，需使用babel-loader将js转化为更被浏览器广泛支持的ES5语法。

    输出管理

    对于单页面应用，只需要在用户第一次请求的时候返回一个HTML文件，而在这个HTML文件中，需要把webpack打包后的js文件引入，使用HtmlWebpackPlugin插件可以自动的替我们完成这项工作。

    开发环境

    在开发环境下，我们一般是在本地服务器上开发，由于需要不断的调试，我们希望打包后还能对应出源文件结构从而定位出bug所在的位置，所以我们需要精确的source map。当我们修改完后还需要等webpack重新打包完后才能看到更改后效果。所以我们需要尽可能短的打包时间。如果能每当我们修改并保存后，webpack就能自动重新打包（只打包修改后的模块）并在浏览器自动加载修改后的结果就更好了。幸运的是，这些webpack都可以替我们做到。
使用webpack-dev-server

webpack-dev-server 提供一个简单的Node服务器，保存后浏览器自动重载。并且可以在配置中开启HMR，只重新构建修改了的模块，极大缩短了重新构建所需的时间。

生产环境

在生产环境中，我们的目标转换为尽可能减少构建后的代码，不需要很精确的source maps。
使用UglifyJSPlugin插件压缩构建后的代码。

指定环境

使用webpack自带的DefinePlugin插件指定生产环境和开发环境。如，在生产环境下的vue会添加很多错误提示和辅助信息使得调试变得更容易些。而在生产环境为了减少包的大小，这些错误提示是不应该包含进来的。

代码分割

有三种常用的分割代码的方法：

  • 通过指定多个入口文件
  • 使用CommonChunkPlugin插件提取出公用的代码
  • 使用import()或require.ensure动态导入模块

    使用缓存

    当用户第一次请求资源并下载到本地后，把资源缓存起来，而当用户再次请求相同的资源时，就可以利用本地缓存的文件。

但是怎么分辨出资源是否没有变化呢，通过资源名字的变化是一个很好的方法。因此，我们可以给打包后的文件名字添加[hash]或[chunkhash]子字段来辨别资源的变化。

在之前除了可以使用CommonsChunkPlugin提取出第三方的库代码外，还可以提取出webpack runtime和module manifest，因为这两个在每次构建的时候都会变化。这样在重新构建的时候引用的第三发库代码的hash值就不会发生变化。

然而，会出现一种情况，当我们添加一个新的模块再次重新构建后，发现提取出的第三方的库代码的hash值也发生的变化。原因是module.id发生的变化。这种情况可以通过NameModulesPlugin或HashedModuleIdsPlugin插件来解决。

JavaScript异步编程

本文旨在介绍JavaScript中异步编程的实现和发展，是从遇到问题解决问题的过程，从最开始的callback到现在的async/await。从历史的顺序介绍事物的发展的过程可以使我们更好的理解。

JavaScript异步

代码是通过CPU一条一条执行的，但如果一条代码需要长时间的等待I/O资源，为了保证代码的顺序执行，CPU就不得不停下来等待。异步就是解决这个问题而提出的。异步是JavaScript的核心特性，js中的异步大多是遇到一个需要长时间等待I/O的任务，这时候js不会停下来去等这个任务完成，而是接着执行接下来的代码，等到这个任务的I/O的完成后再接着执行这个任务。

回调函数

js对异步编程的实现，就是回调函数。回调函数是指，在一个任务等到I/O完成后再去执行的函数。要在等一个任务I/O完成后再去执行的代码就必须放在回调函数里面。但是当有多个异步任务时，就会出现回调函数嵌套回调函数的情况。而且在模块化编程中，回调函数必须在不同的模块之间传递，给代码维护造成的很大的困难。

Promise

Promise是解决callback噩梦而提出的。

首先，Promise是一个对象，它有以下三个状态：

  • pending: 初始状态，成功或失败状态。
  • fulfilled: 意味着操作成功完成。
  • rejected: 意味着操作失败。

Promise的构造函数接收一个executor函数作为参数，在这个executor函数中，可以执行异步代码，executor函数可以接收两个函数，一个是resolve函数，一个是reject函数。异步操作完成后会执行resolve函数，Promise将会由最初的pending状态转变为fulfilled状态。如果在执行异步操作的过程中发生错误，则会执行reject函数，Promise由pending状态转变为rejected状态。

我们将要等异步操作完成后执行的代码放在Promise.then函数中，它可以接收两个函数（分别是onfulfilled和onrejected）作为参数。
当Promise状态变为fulfilled时执行onfulfilled函数，状态变为rejected时执行onrejected函数。

因为Promise.then函数会返回一个新的Promise对象。所以Promise可以链式执行。

之前callback的思想是，如果一个函数A中含有异步操作，那么就必须将等待异步完成后执行的代码放在一个函数B中作为参数传给函数A。函数B是附着在A上的。

用了Promise后含有异步操作的函数返回一个Promise，异步执行完后要执行的函数B是放在Promise.then中的，实现了代码的分离。多重异步嵌套现在变为链式增长。

Generator

Generator函数与普通函数的区别是，它可以暂停执行，交出函数的执行权。调用Generator函数会返回一个内部指针，调用指针的next方法，会移动内部指针（即执行异步任务的第一段），指向第一个遇到的yield语句。next方法的作用是分阶段执行Generator函数。每次调用next方法，会返回一个对象，表示当前阶段的信息（value属性和done属性）value属性是yield语句后面表达式的值，表示当前阶段的值done属性是一个布尔值，表示Generator函数是否执行完毕，即是否还有下一个阶段。

yield语句后一般跟一个Promise对象。

async/await

async函数即是将Generator函数的星号（*）替换成async，将 yield 替换成await，然后自带执行器，所以async函数的执行，与普通函数一样。await跟generator函数中的yeild一样，后面只能跟一个Promise对象。

在Promise中，异步操作成功后会执行resolve函数将Promise状态变为fulfilled。在Promise.then中异步操作成功后要执行的onfulfilled函数在Promise状态变为fulfilled后执行。

而在async函数中，遇到await会等异步完成，异步操作成功后会执行resolve函数将结果返回。然后继续执行await后的代码，不用再在promise.then中执行异步成功后的代码。代码书写起来就是按顺序的，没有其他多余的操作。

最开始的封装

需要注意的是async/await也是基于Promise的。而用Promise处理异步代码，最开始是要new一个Promise对象的。在这个对象中封装了一个异步。那么怎么知道这个异步什么时候完成呢，这里就只能用js原生的callback了。在callback中触发Promise的resolve函数，将Promise的状态变为fulfilled，这样就可以通知Promise链上的代码依次执行了。

  • \l，列举出所有的数据库

  • \dt，进入某个数据库后，列举出所有的表，相当于mysql的show tables
  • \d tablename，查看表的结构，相当于mysql的desc tablename
  • \q，退出登录，相当于mysql的quit

这个文件的主要作用是控制可访问的主机，可访问的用户，可访问哪一个数据库，以及验证的方法，验证方法默认为ident，修改为password即可使用用户名和密码方式访问，由于password是使用明文密码的方式，出于安全性的考虑，可使用md5加密访问

前端框架带来的改变

前端框架带来的改变

web端的兴起

传统的前端开发，是由HTML,CSS和JS来组成的，HTML控制网页结构，CSS改变样式，JS提供和用户的交互功能。web端相比于桌面应用和app，首先是跨平台的，其次，每次app的改版必须是要下载和安装新的软件包的，而频繁的更迭版本带来的用户体验是很差的，而web端相对用户来说只需一个链接，更轻量化和灵活。web端的内容也容易被分享和传播，而且可以被搜索引擎索引，这些也都是app所达不到的。

为什么会出现前端框架

在用web端来实现app的功能时，web网页不再是展示的作用，而更多的承担和用户交互的功能。在web开发中我们使用JS来响应用户的点击，输入等操作，在JS中操作DOM树来更新视图。但对于交互要求很频繁的网页，开发的过程会遇到如下问题：

  • 大量对DOM的操作
  • 重复性地实现相似的功能
  • 代码不能复用

因此，框架的出现就是为了解决这些问题。首先，将数据和视图绑定，视图的更新不再依赖于编写JS代码来手动改变DOM树的方式来改变视图，而是，通过视图和数据绑定后，框架监测数据的变化，来自动更新视图。开发人员现在只需关注对数据的操作，由此带来的视图变化由框架自动来完成。其次，出现了组件，组件是将一些独立的功能和视图模块封装起来，实现了代码的复用，并提高代码的可维护性。

使用框架给开发者带来的变化

在没有框架之前，都是在服务器端把页面渲染好再传到前端显示，使用了框架后，页面渲染在前端完成，后台要做的只是响应前端的数据请求，以json的形式将数据发送给前端即可。

框架和webpack

由于单文件组件的出现，我们将HTML，JS，CSS都写在一个文件中，这个文件会有一个特殊的后缀名，如vue的.vue文件。这样的特殊后缀名的文件浏览器是不能识别的。因此，我们就需要类似webpack之类的工具处理一下，把它当中的内容提取出来。而webpack实际上是一个模块打包工具。采用模块化的思想可以编写复杂的应用，而JS在ES6之前还不能支持模块化。使用webpack可以分析你的项目的结构，找出模块之间的依赖关系，打包为可被浏览器识别的文件。而且，在打包的过程中也可以对项目中的文件做一些处理，比如使用bable将es6的语法转化为es5，一些CSS预处理文件的编译，图片的压缩和转化等。使用webpack可以很大程度的提高构建时的效率，实现前端构建的自动化。

使用webpack-dev-server

在开发的过程中，要不断修改源码并要在浏览器看到效果，每次修改完都使用webpack重新打包一遍，对于大型的项目来说，这个时间时很长的。而且由于要频繁的改动，长时间的等待是无法容忍的。使用了webpack-dev-server，可以自动监听文件的修改，并重新打包。打包后的文件是放在内存当中的，而且可以启用Hot Module Replacement功能，这样打包的时候只会替换修改了的模块。极大的缩短了重新打包的时间。而且它提供了一个小型的node服务器，可以向浏览器发送静态资源，因此，前端项目可以直接运行并在浏览器看到效果。

从微信公众号H5页面生成对应的小程序页面

写在前面：

刚来到公司实习，公司刚开始开发了一个完整的微信公众号的服务，后来出了小程序后，要把微信公众号的h5页面在小程序上也实现，功能和页面的样式都保持和h5一样，基于这样的需求来总结一下微信从h5到小程序的转化过程。

小程序和公众号

从这两者扮演的角色来看，公众号服务于营销和信息传递，而小程序面向产品与服务，因此小程序承担更多和用户交互的功能。从技术角度，h5是一个开放标准规范，而小程序是封闭的，只能在微信里运行，而且有自己独特的标准。

小程序特性

小程序有自己的开发框架

框架提供了自己的视图层描述语言 WXML 和 WXSS，以及基于 JavaScript 的逻辑层框架，并在视图层与逻辑层间提供了数据传输和事件系统，可以让开发者可以方便的聚焦于数据与逻辑上。
其中WXML对应html视图界面，WXSS对应CSS样式，JS框架主要提供了逻辑层和视图层的数据绑定，框架也管理了小程序的页面路由，管理页面的生命周期。开发者只需将页面的数据，方法和声明周期函数注册进框架中，然后由框架来处理剩下的工作。

小程序组件

这里的组件和我们一般在JS框架里说的组件不是同一个概念，这里的组件是视图层的基本组成单元。即和html页面中的元素相对应，不过是在微信这个平台的实现，以达到小程序接近原生APP的体验。两者较大的区别如下 ：

  • WXML语言必须包括开始标签和结束标签
  • 没有div标签，可用view组件代替
  • WXML提供两种文件引入方式，import和include
  • icon可以直接用微信组件默认的图标，默认是iconfont格式的
  • input 的类型，有效值：text, number, idcard, digit, time, date
  • checkbox和radio都单独做成了组件，默认的input只支持输入文本
  • picker组件：从底部弹起的滚动选择器，支持普通、日期和时间等五种选择器
  • navigator支持相对路径和绝对路径的跳转，默认是打开新页面，当前页面打开需要加redirect
