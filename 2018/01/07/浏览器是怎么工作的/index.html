浏览器是怎么工作的

浏览器的主要组成部分

  1. 用户界面-包括地址栏，向前向后按钮，书签菜单等。
  2. 浏览器引擎-查询和操作渲染引擎的接口。
  3. 渲染引擎-作用是显示请求的资源，比如，如果请求的内容是HTML，那么它就负责解析HTML和CSS然后把解析的结果显示到屏幕上。
  4. 网络模块-发送网络请求，比如HTTP请求。
  5. 后端UI-绘制基础元素。它提供一个与平台无关的通用接口，在往下就是操作系统提供的用户接口方法。
  6. JavaScript解释器-解析和执行JavaScript代码。
  7. 数据存储-浏览器需要存各种数据在硬盘上，比如，cookies。

渲染引擎

基本流程

从网络层得到文档后，渲染引擎会先从解析HTML文档开始，把标签翻译为DOM节点，生成一个内容树(content tree)。解析CSS，得到的样式信息和之前的内容树结合生成渲染树(render tree)。渲染树经过布局(layout)处理，即确定每一个节点在屏幕上的位置后，就开始绘制(painting)，生成最终的页面。

处理scirpts和CSS的顺序

Scripts

开发者会期望在解析时遇到标签后，立刻解析和执行scripts（因为script执行可能会改变DOM结构）。同时，会停止解析文档直到script执行完后。如果script是引用外部文件，解析也会停止，直到外部文件被下载和执行完毕。但是开发者可以在script标签上添加defer属性，这样script会在文档解析完后再执行。另外，async属性使得外部script的解析和执行是在一个新的线程上，不停止文档的解析。

解析优化

当执行script，会另起一个线程解析剩余的文档，加载外部引用的的资源。注意的是，它只会解析外部引用文件，如，外部scripts，外部CSS和图片等，不会修改DOM树。

CSS

因为CSS不改变DOM树的结构，看起来没有理由要等CSS解析而停止文档的解析。但会存在一个问题，如果scripts在文档解析阶段需要请求样式信息，如果CSS还没被加载和解析，那么scirpts就会因为得不到结果而产生一系列的问题。看起来极端的情况却是经常出现的。Firefox会阻塞所有scripts当还有CSS正在被加载和解析时。Webkit只当scripts试图访问特定的样式信息时阻塞他们。
